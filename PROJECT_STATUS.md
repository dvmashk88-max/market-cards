## Что сейчас работает

### Railway deployment

- Основная витрина разворачивается на Railway из того же публичного GitHub-репозитория.
- Railway использует корень репозитория как Root Directory, поскольку сборке требуются корневые `pnpm-workspace.yaml`, lockfile и workspace-зависимости.
- Build Command: `pnpm --filter @workspace/antarctic-market run build`.
- Start Command: `pnpm --filter @workspace/antarctic-market run serve`.
- Команда запуска поднимает Vite preview на `0.0.0.0` и использует порт, переданный Railway через переменную `PORT`.
- Healthcheck Railway выполняется по пути `/`. Настроен перезапуск при ошибке с ограниченным числом повторов.
- Проверенное состояние сервиса: deployment `SUCCESS`, replica работает, сервис не остановлен.

### GitHub Auto Deploy

- Railway связан с репозиторием `dvmashk88-max/market-cards` и веткой `main`.
- После push нового коммита в `origin/main` Railway автоматически получает изменения, выполняет production build и заменяет опубликованную версию после успешного завершения сборки и healthcheck.
- Отдельный ручной перенос файлов на Railway для обычных обновлений витрины не требуется.

### Публичный URL

- Рабочая витрина доступна по адресу <https://market-cards-production.up.railway.app>.
- Это текущий основной публичный домен Railway. Пользовательский домен отдельно не подключался.

### Переходы в MAX и Telegram

- Кнопка «Перейти в MAX» в hero-блоке ведёт на <https://max.ru/id6321431962_1_bot>.
- Кнопка «Перейти в Telegram» рядом с ней ведёт на <https://t.me/marketcards163bot>.
- Ссылки открываются в новой вкладке безопасным способом. Дублирующей MAX-кнопки в панели заказа больше нет.

### Production build и typecheck

- Установка зависимостей через `pnpm install` проходит успешно.
- Корневая команда `pnpm run typecheck` успешно проверяет TypeScript-проекты монорепозитория, включая основную витрину, API-сервер, mockup workspace и служебные скрипты.
- Production build основной витрины успешно выполняется командой `pnpm --filter @workspace/antarctic-market run build`.
- Railway использует ту же целевую production-сборку, поэтому локальная и удалённая процедуры согласованы.

### SPA routing

- Витрина является SPA на React и использует Wouter для клиентской маршрутизации.
- Работают главная страница и информационно-правовые маршруты: `/`, `/oferta`, `/privacy`, `/personal-data`, `/terms`, `/refund`.
- Vite preview возвращает приложение для прямых переходов по SPA-маршрутам; неизвестные клиентские маршруты обрабатываются предусмотренным приложением fallback-экраном.

### Текущая архитектура

- Репозиторий является pnpm-монорепозиторием.
- Основной frontend: `artifacts/antarctic-market` (`@workspace/antarctic-market`). Технологии: React 19, TypeScript 5.9, Vite 7, Tailwind CSS 4, Wouter, Framer Motion и существующий набор UI-компонентов.
- Отдельный `artifacts/mockup-sandbox` используется как вспомогательный workspace и не является публикуемой витриной.
- Backend `artifacts/api-server`, библиотека базы данных и сгенерированный API-клиент сохранены в репозитории, но не участвуют в текущем Railway deployment витрины.
- Текущий frontend не вызывает backend и внешние товарные API. Маршрут `/api/healthz` относится к сохранённой backend-заготовке и для работы витрины не нужен.

## Следующий этап

Следующая большая задача — превратить текущую frontend-витрину в полноценный интернет-магазин цифровых товаров. Целевая схема взаимодействия:

```text
Frontend (Railway)
        ↓
Backend
        ↓
FazerCards API
```

Frontend не должен обращаться к FazerCards API напрямую. Backend должен стать доверенной границей системы: хранить секреты, проверять цены и товары, создавать заказы, управлять оплатой, запрашивать цифровой товар у поставщика и безопасно отдавать покупателю только разрешённый результат.

### 1. Зафиксировать контракт с FazerCards API

- Получить актуальную документацию FazerCards API, тестовый доступ и отдельные production-учётные данные.
- Определить способ авторизации, базовые URL для test/production, лимиты запросов, поддерживаемые валюты, идентификаторы товаров и номиналов.
- Описать методы получения каталога и доступности, расчёта цены, создания покупки, получения результата и проверки статуса.
- Проверить, поддерживает ли FazerCards webhooks. Если нет, спроектировать безопасный polling с ограничением частоты, таймаутами и остановкой после терминального статуса.
- Зафиксировать формат ошибок, правила повторного запроса и случаи, когда повтор операции может привести к двойной покупке.

### 2. Спроектировать backend как отдельный сервис

- Сохранённую backend-заготовку можно развить либо заменить осознанно после фиксации API-контракта; удалять её до этого не требуется.
- Разделить backend на модули каталога, ценообразования, заказов, платежей, выдачи, уведомлений и интеграции FazerCards.
- Определить публичные endpoint-ы frontend → backend, например: каталог, карточка товара, расчёт заказа, создание checkout, получение статуса заказа и безопасное получение результата.
- Валидировать все входные данные на сервере. Никогда не принимать итоговую цену, статус оплаты или факт выдачи как доверенные данные от frontend.
- Настроить CORS только для разрешённых доменов витрины, rate limiting, безопасные заголовки, ограничения размера запросов и структурированную обработку ошибок.
- Хранить `FAZERCARDS_API_KEY`, секреты платежей, `DATABASE_URL`, SMTP/API-ключи почты и другие чувствительные значения только в Railway Variables соответствующего backend-сервиса.

### 3. Создать модель данных и миграции

- Подключить production PostgreSQL и оформить версионируемые миграции.
- Создать сущности для товаров и снимков цен, покупателей, заказов, позиций заказа, платежных попыток, операций FazerCards, выданных кодов/ссылок, событий webhook и журнала аудита.
- Определить уникальные идентификаторы и idempotency keys, чтобы повтор запроса, webhook или фоновой задачи не создавал повторную оплату либо повторную покупку у поставщика.
- Зафиксировать жизненный цикл заказа, например: `draft` → `awaiting_payment` → `paid` → `fulfilling` → `delivered`, а также ветви `failed`, `cancelled`, `refunded` и `manual_review`.
- Настроить резервное копирование, сроки хранения персональных данных и процедуру восстановления.

### 4. Реализовать каталог и ценообразование

- Синхронизировать товары FazerCards с внутренними товарными сущностями, не связывая frontend напрямую с нестабильными идентификаторами поставщика.
- Кэшировать каталог и доступность с контролируемым TTL, чтобы витрина оставалась быстрой и не превышала лимиты FazerCards.
- Рассчитывать конечную цену только на backend с учётом закупочной цены, комиссии, маржи, валюты и правил округления.
- Перед оплатой повторно подтверждать доступность и цену. Если цена изменилась, возвращать frontend новую котировку, а не списывать неожиданную сумму.
- Предусмотреть возможность временно скрывать товар или номинал без изменения frontend-кода.

### 5. Реализовать checkout и жизненный цикл заказа

- Frontend отправляет выбранные позиции и email на backend; backend проверяет товары, формирует неизменяемую котировку и создаёт заказ.
- Backend создаёт платёжную сессию и возвращает frontend только безопасные данные для перехода к оплате.
- После подтверждённой оплаты backend запускает выдачу через FazerCards. Одного редиректа пользователя со страницы оплаты недостаточно: источником истины должен быть подписанный webhook платёжного провайдера.
- Все переходы статусов должны быть атомарными, повторяемыми без двойного эффекта и отражаться в журнале событий.
- Для долгих операций потребуется очередь или фоновые задачи с retry/backoff, dead-letter/manual-review сценарием и защитой от повторной выдачи.

### 6. Подключить оплату

- Выбрать платёжного провайдера с поддержкой нужной юрисдикции, валюты, чеков и возвратов.
- Проверять подпись каждого webhook, сохранять необработанное событие и обеспечивать идемпотентную обработку.
- Сверять сумму, валюту и идентификатор заказа до перевода заказа в статус `paid`.
- Реализовать отмену, полный или частичный возврат и ручную сверку зависших платежей.
- Не хранить данные банковских карт в приложении; использовать размещённую платёжную страницу или безопасные компоненты провайдера.

### 7. Реализовать выдачу и email-доставку

- Получать цифровой товар только на backend после достоверного подтверждения оплаты.
- Шифровать чувствительный результат в базе либо хранить его минимально необходимое время. Исключить попадание кодов в логи, аналитику, URL и сообщения об ошибках.
- Отправлять покупателю письмо через транзакционный почтовый сервис, с шаблонами для успешной выдачи, задержки и ошибки.
- Добавить защищённую страницу статуса заказа с одноразовым или ограниченным по времени токеном; не использовать предсказуемые идентификаторы.
- Предусмотреть повторную отправку письма, ручную проверку и подтверждение фактической выдачи.

### 8. Подключить frontend к backend

- Заменить демонстрационные данные реальным каталогом backend, сохранив текущий визуальный стиль.
- Добавить состояния загрузки, отсутствия товара, изменения цены, ошибки API, ожидания оплаты и выполнения заказа.
- Реализовать настоящую корзину и checkout, серверную валидацию email и понятное подтверждение условий покупки.
- Не раскрывать секреты и внутренние административные URL в клиентском bundle. Frontend должен знать только публичный URL backend и публичные идентификаторы платёжного провайдера, если они действительно предназначены для браузера.
- Сохранить прямые ссылки MAX и Telegram как каналы связи, не смешивая их с автоматизированным процессом заказа.

### 9. Добавить кабинет покупателя и историю заказов

- Выбрать модель доступа: полноценная учётная запись либо безопасные magic links для просмотра заказа без пароля.
- Реализовать просмотр статусов и истории, повторную отправку результата в допустимых случаях и обращение в поддержку.
- Разделить права покупателя, оператора и администратора. Все проверки авторизации выполнять на backend.
- Подготовить процессы удаления/экспорта персональных данных и актуализировать юридические документы под фактическую обработку данных.

### 10. Создать административные и операционные инструменты

- Добавить защищённую административную панель для просмотра заказов, платежей, выдач, ошибок и расхождений.
- Реализовать управление публикацией товаров, наценкой, лимитами и временной недоступностью.
- Опасные ручные операции — возврат, повторная выдача, изменение статуса — должны требовать явного подтверждения и записываться в аудит.
- Подготовить инструкции для поддержки: что делать при оплаченном, но не выданном заказе, задержанном webhook, ошибке FazerCards или спорном платеже.

### 11. Обеспечить качество, безопасность и наблюдаемость

- Добавить unit-тесты расчёта цены и переходов статусов, integration-тесты backend/DB/FazerCards adapter и end-to-end сценарии checkout.

- Добавить структурированные логи без секретов и цифровых кодов, метрики, трассировку, error tracking и алерты на ошибки оплаты/выдачи.
- Ввести отдельные staging и production окружения, разные ключи и базы, проверяемые миграции и контролируемый rollout.
